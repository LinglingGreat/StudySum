### Anthropic Engineering：《Scaling Managed Agents: Decoupling the brain from the hands》

Anthropic 发布了 Managed Agents 背后的架构演进——核心思路是把"大脑"（Claude + harness）与"双手"（sandbox 和 tools）以及"记忆"（session log）三者彻底解耦。

**为什么要解耦？** 最初的设计把所有组件塞进同一个容器，这让基础设施变成了"宠物"而非"牛群"：容器挂了，整个 session 就丢了；调试时必须进入同时存有用户数据的容器，几乎无法操作；客户想把 Claude 接入自己的 VPC，就得把网络打通到 Anthropic 这边。

**解耦之后：**
- Harness 移出容器，把 sandbox 当作普通工具调用：`execute(name, input) → string`。容器挂了，harness 捕获到 tool call 错误传回给 Claude，重启一个新容器即可。
- Session log 独立存储在容器外部，harness 挂了也可以通过 `wake(sessionId)` 恢复，无状态重启。
- 安全边界明确：sandbox 里运行的代码永远接触不到 credentials。Git token 在初始化时直接 clone 进去，OAuth token 通过 vault + proxy 管理，harness 本身从不接触任何凭证。
- **性能提升显著：** p50 首 token 延迟（TTFT）下降约 60%，p95 下降超过 90%。原因是容器只在真正需要时才被 brain 按需调用，不再每个 session 都预先付出容器启动成本。
- **多 brain、多 hand 扩展：** brain 之间可以互相传递 hand，支持并行大规模扩展。

文章还讨论了 session 作为"上下文对象"的设计——它不是 Claude 的 context window，而是一个存在 context window 之外、可以被 brain 按需读取的事件流，避免了传统 compaction 方案不可逆丢弃信息的问题。

https://www.anthropic.com/engineering/managed-agents

### No Priors：《SAP: Bringing the 'Operating System' of a Company into the AI Era with CTO Philipp Herzig》

**核心要点：** AI 对企业的改变不是技术升级，而是商业模式转型——但大多数创业者严重低估了规模带来的复杂性。

SAP CTO Philipp Herzig 把企业 AI 的改变分为三层：UI 层（从固定界面转向 generative UI，系统主动推送洞察而非等用户点击）、业务流程层（agents 把结构化与非结构化世界融合，从"软件即服务"走向"服务即结果"）、数据层（统一的语义数据模型是 AI 发挥作用的前提）。

**最反直觉的观点：LLM 不适合做预测。** Herzig 明确指出，大语言模型在非结构化世界表现出色，但对于企业真正需要的预测性分析（需求预测、现金流预测、付款延迟预测），LLM 从根本上就不适合，因为它们是语言模型而非统计模型。SAP 为此研发了两年，提出了 RPT-1（Relational Pretrained Transformer），把 transformer 架构用于关系型表格数据，让"小样本高精度预测"成为可能，而无需专业数据科学家。

**规模是最大的工程挑战，不是 AI 本身。** 他举例：10 个文档的 RAG 很容易，1000 个文档就是深度工程挑战；10 个 MCP API 没问题，SAP 有 20,000 个 API，context bloat 问题让整合难度指数级上升。

**关于 evals 的洞察：** 他把写 eval 比作 TDD（测试驱动开发）的回归。以前没人真的写 eval，就像以前没人真的写测试——因为先写代码更有趣，而且需求本身就是混乱的。但现在代码的生成已经高度自动化，真正的挑战变成了"如何描述你想要的结果"，而 eval 正是这种描述的载体。

**商业模式转型：** SAP 正在从按席位收费走向消费量计费，下一步是结果计费（类似 Sierra 的做法）。但企业客户目前仍需要可预测性，所以是混合模式，并随客户成熟度逐步演进。
