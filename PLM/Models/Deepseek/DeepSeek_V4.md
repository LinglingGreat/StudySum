## 一、核心概览

DeepSeek-V4 系列包含两款 Mixture-of-Experts (MoE) 语言模型：

| 模型 | 总参数 | 激活参数 | 特点 |
|------|--------|---------|------|
| **DeepSeek-V4-Pro** | 1.6T | 49B | 旗舰版，最强性能 |
| **DeepSeek-V4-Flash** | 284B | 13B | 高效版，极高性价比 |

两个模型均支持 **100万 token 上下文长度**，模型权重已在 HuggingFace 开源：https://huggingface.co/collections/deepseek-ai/deepseek-v4

## 二、核心架构创新

### 1. 混合注意力架构（CSA + HCA）

这是 V4 最重要的创新，旨在解决超长上下文场景下的计算瓶颈：

- **CSA（Compressed Sparse Attention）**：将每 m 个 token 的 KV cache 压缩为 1 个条目，再通过 Lightning Indexer 进行 top-k 稀疏选择，只对选中的压缩 KV 执行核心注意力。
- **HCA（Heavily Compressed Attention）**：更激进的压缩，每 m'（≫m）个 token 压缩为 1 个条目，不做稀疏选择，直接做全量密集注意力。
- **交替混合配置**：CSA 和 HCA 交替排列，兼顾效率与表达能力。

**效率提升（对比 DeepSeek-V3.2，100万 token 场景）**：

| 模型 | 推理 FLOPs | KV Cache 大小 |
|------|------------|----------------|
| V4-Pro | 27% (仅为 V3.2 的 27%) | 10% |
| V4-Flash | 10% | 7% |

### 2. 流形约束超连接（mHC）

替代传统残差连接：

- 将残差映射矩阵约束在**双随机矩阵流形**（Birkhoff 多胞形）上，保证谱范数 ≤ 1，提升数值稳定性
- 引入动态参数生成（输入依赖 + 静态分量），通过 Sinkhorn-Knopp 算法投影到约束流形
- 拓展残差流宽度（n_hc 倍），与隐藏层维度解耦，以极小计算开销提供额外的模型容量

### 3. Muon 优化器

- 对大部分参数采用 Muon 优化器（基于 Newton-Schulz 迭代的正交化优化），相比 AdamW 收敛更快、训练更稳定
- Embedding、预测头、RMSNorm 等少数模块仍用 AdamW
- 采用**混合 Newton-Schulz 迭代**（前8步快速收敛 + 后2步精调奇异值至1）
- 引入 **Nesterov 动量技巧**和更新矩阵 RMS 重缩放

## 三、基础设施优化

1. **细粒度专家并行通信-计算重叠**：将 MoE 层的 Dispatch/Linear-1/Linear-2/Combine 四个阶段融合为统一流水线，理论上可获 1.92× 加速
2. **TileLang DSL**：用于平衡开发效率与运行时性能的内核开发领域专用语言
3. **FP4 量化感知训练**：对 MoE 专家权重和 Indexer QK 路径采用 FP4 精度，为未来硬件的 1/3 效率提升做准备
4. **KV Cache 混合精度存储**：RoPE 维度用 BF16，其余用 FP8，KV cache 大小相比纯 BF16 减少近半
5. **训练框架扩展**：扩展自动微分框架支持张量级 Checkpointing、两阶段上下文并行、Muon 的混合 ZeRO 策略
6. **磁盘 KV Cache 存储**：支持高效的共享前缀复用

## 四、预训练

### 训练配置

- **数据规模**：V4-Flash 训练 32T tokens，V4-Pro 训练 33T tokens
- **序列长度课程**：4K → 16K → 64K → 1M 渐进式扩展
- **稀疏注意力热身**：前 1T tokens 用密集注意力，之后引入稀疏注意力；先短时间预热 Lightning Indexer，再全面启用
- **批大小调度**：从较小值逐步增加到 75.5M tokens 并保持

### 路由策略改进

- 将激活函数从 Sigmoid 改为 **Sqrt(Softplus)**
- 引入 **Hash 路由**（前几层 MoE 用哈希函数决定目标专家）
- **预测性路由（Anticipatory Routing）**：用历史参数计算路由索引，解耦主干网络与路由网络的同步更新，有效防止 loss spike，额外开销仅约 20%
- **SwiGLU Clamping**：限制数值范围，消除异常值，提升训练稳定性

### 负载均衡

- 无辅助损失的平衡策略 + 序列级轻微平衡损失（权重 0.0001）
- MTP 损失权重：大部分训练为 0.3，学习率衰减阶段为 0.1

## 五、后训练（Post-Training）

后训练流水线整体沿用 DeepSeek-V3.2 的框架，但关键区别在于：**用 On-Policy Distillation (OPD) 完全取代了混合强化学习阶段**。

### 5.1 后训练流水线

#### 5.1.1 领域专家独立培养（Specialist Training）

每个领域专家（数学、编程、Agent、指令遵循等）依次经过两个阶段：

**阶段一：监督微调（SFT）**
- 在高质量、领域特定的数据上进行监督微调，建立基础能力

**阶段二：强化学习（RL）**
- 采用 **GRPO（Group Relative Policy Optimization）** 算法
- 由针对特定成功标准定制的奖励模型引导

**三种推理努力模式（Reasoning Efforts）**

| 模式 | 特点 | 典型场景 | 响应格式 |
|------|------|----------|----------|
| **Non-think** | 基于直觉的快速响应 | 日常任务、紧急反应、低风险决策 | 直接输出（无 think 标签） |
| **Think** | 有意识的逻分析，较慢但更准确 | 复杂问题求解、规划、中风险决策 | `<think>` 思考 tokens `</think>` + 摘要 |
| **Think Max** | 推理推至极限，最慢但最强 | 探索模型推理能力边界 | 特殊系统提示 + `<think>` 思考 tokens `</think>` + 摘要 |

- 每种模式在 RL 训练中使用不同的**长度惩罚**和**上下文窗口**，导致推理输出 token 长度不同
- Think Max 模式在系统提示开头注入特殊指令："Reasoning Effort: Absolute maximum with no shortcuts permitted"，要求模型彻底分解问题、严格测试所有路径

**生成式奖励模型（Generative Reward Model, GRM）**
- 传统难验证任务依赖 RLHF + 标量奖励模型 + 大量人工标注
- DeepSeek-V4 改用**生成式奖励模型**：演员网络本身充当 GRM，直接对策略轨迹进行评估
- 模型的评估能力与生成能力联合优化，用极少的人工标注实现优异性能
- 易验证任务仍使用基于规则的验证器或测试用例

**工具调用模式（Tool-Call Schema）**
- 引入新的工具调用模式，使用特殊 `<|DSML|>` token + XML 格式
- XML 格式有效减少转义失败、降低工具调用错误
- 工具定义包含在 `<|DSML|tool_calls>` 块中，参数类型区分 string 和非 string

**交错思考（Interleaved Thinking）**
- DeepSeek-V3.2 的缺陷：每次新用户消息到达时丢弃历史思考内容，导致模型需从头重建解题状态
- V4 改进（工具调用场景）：**保留完整推理历史跨越所有轮次**，包括跨越用户消息边界，支持长程 Agent 任务中的连贯思维链
- V4 保留原策略（通用对话场景）：新用户消息到达时丢弃旧推理内容

**Quick Instruction（快速指令）**
- 聊天场景中，许多辅助任务（如判断是否触发搜索、意图识别等）需要在生成响应前执行
- 传统做法：用单独的小模型处理，无法复用已有 KV cache，需要冗余 prefill
- Quick Instruction：直接将一组特殊 token 附加到输入序列，每个 token 对应一个辅助任务，**完全复用已计算的 KV cache**，避免冗余 prefill
- 支持的辅助任务：

| 特殊 Token | 功能描述 |
|-----------|----------|
| `<|action|>` | 判断用户提示是否需要网络搜索 |
| `<|title|>` | 在首次助手响应后生成对话标题 |
| `<|query|>` | 为用户提示生成搜索查询 |
| `<|authority|>` | 分类用户提示对来源权威性的需求 |
| `<|domain|>` | 识别用户提示的领域 |
| `<|extracted_url|>` / `<|read_url|>` | 判断提示中的 URL 是否应被获取和读取 |

#### 5.1.2 在策略蒸馏（On-Policy Distillation, OPD）

**核心思想**：让学生模型（统一模型）从教师模型在自己生成轨迹上的输出分布中学习，通过反向 KL 散度损失对齐：

```
L_OPD(θ) = Σ w_i · D_KL(π_θ || π_Si)
```

- `w_i` 为每个专家的权重（按专家重要性分配）
- 在策略学习确保学生 π_θ 从与当前任务上下文相关的专家中学习（如数学任务对齐数学专家）
- 超过 **10 个**覆盖各领域的教师模型参与蒸馏，将物理上分离的专家权重通过 logits 级对齐整合到统一参数空间

**全词表蒸馏（Full-Vocabulary OPD）**
- 先前工作：将全词表 KL 损失简化为每个 token 位置的 token 级 KL 估计，复用 RL 框架——但梯度估计方差大，训练不稳定
- DeepSeek-V4 采用**全词表 logits 蒸馏**：保留完整的 logits 分布计算反向 KL 损失，梯度估计更稳定，蒸馏更忠实
- 挑战：词表大小 |V| > 100k，即使写入磁盘也 prohibitive
- 解决方案：在前向传播中仅缓存教师最后一层隐藏状态到集中式缓冲区；训练时检索这些缓存状态，通过对应预测头模块动态重建完整 logits

### 5.2 后训练基础设施

#### 5.2.1 FP4 量化集成
- 对 rollout 和所有仅推理的前向传播应用 **FP4 (MXFP4) 量化**，减少内存流量和采样延迟
- Rollout 和推理阶段直接使用原生 FP4 权重
- 训练步骤中，FP4 量化通过**无损 FP4→FP8 反量化**模拟，无缝复用现有 FP8 混合精度框架
- Indexer QK 路径的 index scores 从 FP32 量化为 BF16，实现 top-k 选择器 2× 加速，同时保持 99.7% 的 KV 条目召回率

#### 5.2.2 高效教师调度（全词表 OPD）
- 支持**无界数量的教师**（每个可能包含数万亿参数）
- 所有教师权重卸载到集中式分布式存储，在前向传播中按需加载，采用 **ZeRO-like 参数分片**缓解 I/O 和 DRAM 压力
- 按教师索引对训练样本排序调度，确保每个 mini-batch 中每个不同教师头仅加载一次，任意时刻最多一个教师头驻留在设备内存中
- 所有参数和隐藏状态加载/卸载操作在后台异步执行，不阻塞关键路径计算
- 使用专用 **TileLang 内核**计算教师和学生的精确 KL 散度，加速计算并减少动态内存分配

#### 5.2.3 可抢占的容错 Rollout 服务
- GPU 集群采用集群级抢占式任务调度器，任何运行中的任务随时可能被抢占
- 实现 **token 级预写日志（WAL）**：每次为新请求生成一个 token，立即追加到该请求的 WAL 中
- 抢占时：暂停推理引擎，保存未完成请求的 KV cache
- 恢复时：使用持久化的 WAL 和保存的 KV cache 继续解码
- 即使发生致命硬件错误，也可使用 WAL 中持久化的 token 重新运行 prefill 阶段重建 KV cache
- 关键：从零重新生成未完成请求在数学上是不正确的（短响应更容易在中断中存活，导致模型倾向生成更短序列）；WAL 方法避免了此问题

#### 5.2.4 百万 Token 上下文的 RL 框架扩展
- Rollout 阶段：采用可抢占的容错 rollout 服务
- 推理和训练阶段：将 rollout 数据格式拆分为轻量级元数据和重型 per-token 字段
- 数据调度时，可加载整个 rollout 数据的元数据以执行全局洗牌和打包布局计算
- 重型 per-token 字段通过共享内存数据加载器加载，消除节点内数据冗余，并在 mini-batch 粒度立即释放，大幅降低 CPU 和 GPU 内存压力
- 设备上 mini-batch 数量根据工作负载动态确定，在计算吞吐量和 I/O 重叠之间高效权衡

#### 5.2.5 Agentic AI 沙箱基础设施（DSec）

构建生产级沙箱平台 **DeepSeek Elastic Compute (DSec)**，由三个 Rust 组件组成（API 网关 Apiserver、每主机代理 Edge、集群监视器 Watcher），通过自定义 RPC 协议互联，在 3FS 分布式文件系统上水平扩展。单个 DSec 集群可管理 **数十万个并发沙箱实例**。

**四大设计考量**：
1. Agent 工作负载高度异构（无状态函数调用 → 完整软件工程流水线）
2. 环境镜像数量多、体积大，但需快速加载并支持迭代定制
3. 高密度部署需要高效的 CPU 和内存利用率
4. 沙箱生命周期必须与 GPU 训练调度协调（抢占 + 基于 checkpoint 的恢复）

**四种执行子系统，统一接口（libdsec Python SDK）**：

| 子系统 | 基础技术 | 适用场景 |
|--------|-----------|----------|
| **Function Call** | 预唤醒容器池 | 无状态调用，消除冷启动开销 |
| **Container** | Docker 兼容 + EROFS 按需加载 | 高效镜像组装 |
| **microVM** | Firecracker | VM 级隔离，安全敏感的高密度部署 |
| **fullVM** | QEMU | 支持任意客户操作系统 |

**核心技术特性**：
- **分层存储快速加载**：基础镜像和文件系统提交存储为 3FS 支持的只读 EROFS 层，元数据本地可用，数据块按需从 3FS 获取；microVM 使用 overlaybd 磁盘格式（只读基础层在 3FS 上共享，写入到本地写时复制层）
- **高密度并发优化**：缓解虚拟环境中重复页缓存占用，应用内存回收实现安全超卖；缓解容器运行时中的 spinlock 争用，降低每沙箱 CPU 开销
- **轨迹日志与抢占安全恢复**：为每个沙箱维护全局有序轨迹日志，持久记录每个命令调用及其结果；支持客户端快进（抢占后重放缓存结果加速恢复）、细粒度溯源、确定性重放

- V4-Pro 还提供 **Max 推理努力模式**（DeepSeek-V4-Pro-Max），通过扩展推理 token 进一步提升性能。

## 六、评估结果

### 知识问答

- V4-Pro-Max 在 SimpleQA、Chinese-SimpleQA 上显著超越领先开源模型
- 与 Gemini-3.1-Pro 的差距大幅缩小，但仍略逊一筹
- 教育知识（MMLU-Pro、HLE、GPQA）方面略有领先

### 推理能力

- V4-Pro-Max 在标准推理基准上优于 GPT-5.2 和 Gemini-3.0-Pro
- 略逊于 GPT-5.4 和 Gemini-3.1-Pro，差距约 **3-6 个月**的技术代差
- V4-Flash-Max 在分配更大思考预算后，推理性能可与 V4-Pro-Max 媲美

### Agent 能力

- 公开基准上与 Kimi-K2.6、GLM-5.1 等领先开源模型持平
- 内部评估超越 Claude Sonnet 4.5，接近 Opus 4.5 水平
- 略逊于前沿闭源模型

### 长上下文

- 在 100万 token 上下文窗口下，在合成和真实用例中均表现强劲
- **在学术基准上超越 Gemini-3.1-Pro**

### 效率对比（关键亮点）

| 指标 | V4-Pro vs V3.2 | V4-Flash vs V3.2 |
|------|----------------|-------------------|
| 单 token 推理 FLOPs（1M 上下文） | 27% | 10% |
| KV Cache 大小（1M 上下文） | 10% | 7% |

相比 BF16 GQA8 基线（常见 LLM 注意力配置），V4 系列在 1M 上下文下的 KV cache 大小可降至约 **2%**。

## 七、总结

DeepSeek-V4 的核心贡献在于**打破了超长上下文的效率瓶颈**：通过 CSA+HCA 混合注意力、mHC 超连接、Muon 优化器三大架构创新，在保持顶级性能的同时，将百万 token 上下文的推理成本降低到此前的 1/10 以下。这使得**测试时扩展（test-time scaling）**和**长程任务**（复杂 Agent 工作流、大规模跨文档分析）变得切实可行。

V4-Flash 尤其值得关注——以 13B 激活参数实现了接近旗舰模型的推理能力，是极具性价比的选择。
